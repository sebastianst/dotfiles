# User Preferences

## Version Control

- If in a JJ repository (`jj root` succeeds), even if colocated with git, use `jj` instead of git:
  - `jj status` and `jj diff --git` instead of `git status` and `git diff`
  - `jj f` to fetch
  - `jj commit -m "msg"` to commit
  - `jj bookmark create <name> -r @-` to create bookmarks
  - `jj p-` to push (pushes `@-`)
  - `jj squash` to amend changes into the previous commit
  - When creating PRs with `gh pr create`, specify `--head <bookmark-name>` since jj doesn't use git branches
- Prefix branch/bookmark names with `seb/`
- Commit descriptions should have the form `<scope>: <desc>`

## Code Style

- Only add non-trivial code comments that genuinely help understanding
- Avoid comments that merely restate what the code does (e.g., "// Capture config values for use in the closure")
- Good comments explain *why*, not *what*

## PR Reviews

- Before addressing comments from AI reviewers (almanax-ai, etc.), ask whether they are relevant
- This gives the user a chance to dismiss false positives at the PR before making unnecessary changes

## GitHub Issues and Artifacts

- Always identify AI-generated artifacts on GitHub with a "🤖 *Generated by Claude Code*" note at the end
- When responding to comments/discussions, start with "🤖 *Claude says:* "
- When creating GitHub issues with sub-issues, use the GitHub sub-issues API to link them as proper sub-issues (not just text references)
- Include GitHub permalinks (with commit hash) for all file path references in issues
